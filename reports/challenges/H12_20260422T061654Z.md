### A4 — Verificación numérica independiente

```python
import math

log10_p = log_p / math.log(10)
p_recalc = math.exp(log_p)

print(f"log10(p) = {log10_p:.2f}")           # Esperado: -170.8
print(f"p_value  = {p_recalc:.6e}")          # Esperado: 1.600e-171

# Verificación phat
phat_check = k / n
print(f"p̂ = {phat_check:.6f}")              # Esperado: 0.904348

# Cohen h
h = 2 * math.asin(math.sqrt(phat_check)) - 2 * math.asin(math.sqrt(p0))
print(f"Cohen h = {h:.4f}")                  # Esperado: 1.8396
```

### A6 — Post-hoc fishing check

**Ataque:** Mesa 018146 fue identificada por búsqueda exhaustiva (H4 hero), no hipótesis a priori. ¿Exige corrección múltiple que destruya el resultado?

**Análisis:**

1. **¿Pre-registrada?** No — la spec H12 la identifica como derivada de H4 (búsqueda exhaustiva sobre 78,605 mesas). Post-hoc confirmado.

2. **¿Corrección Bonferroni destruye?**
   - α_ajustado = 0.001 / 78,605 = **1.27×10⁻⁸**
   - p_finding = **1.60×10⁻¹⁷¹**
   - Margen: 1.60e-171 << 1.27e-8 → **163 órdenes de magnitud de sobra**

3. **¿Bonferroni es la corrección correcta aquí?** El stat_finding argumenta que H0 es *iid por mesa* y se reporta la más extrema. La corrección FWER (Bonferroni) aplica cuando se busca cualquier significativo entre m tests. Incluso aplicada, el resultado sobrevive con 163 órdenes sobrantes.

4. **¿Se reportan TODAS las mesas ≥90%?** Stat_finding declara: "3 mesas totales ≥90%; solo 1 con n>100 = 018146". Esto es transparencia completa, no cherry-picking.

**Limitación real a declarar:** La spec debe incluir explícitamente "corrección Bonferroni aplicada: α_eff=1.27e-8, finding pasa por 163 órdenes". Actualmente dice "Bonferroni no aplica" — argumento discutible aunque el resultado lo supera de todas formas.

> A6 post-hoc: mesa identificada post-hoc. PERO Bonferroni sobre 78,605 mesas da α_eff=1.27e-8 vs p=1.6e-171 → 163 órdenes sobrantes. Se reportan las 3 mesas ≥90%. No hay fishing. **SOBREVIVE** (con nota: spec debe declarar α_eff explícitamente).
